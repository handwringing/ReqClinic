# 需求问诊室发布准入与设计评审记录

> 文档状态：Release Readiness Review v0.1  
> 更新日期：2026-07-06  
> 验收范围：除 Edge 白块专项外的六项长线收口：三功能浏览器验收、正式项目、表达训练、报告质量、Agent + Skill 审计、案例矩阵。

## 1. 结论

当前版本达到“可演示、可设计评审、可继续真实试用”的准入标准。

本轮没有发现 P0 / P1 阻塞问题。已修复两个 P2 问题：

1. 表达训练移动端入口把固定情境案例压到自定义输入后面，导致首屏主次不清。现已改为移动端先显示固定情境，再显示自定义输入说明。
2. 正式项目详细报告重复展示项目描述、角色、材料和约束，并出现用户回答归因到错误地图节点的问题。现已改为项目起点信息只集中展示一次，用户回答按问题语义归入对应节点，候选方案渲染为真实表格。

剩余 P3 观察：

1. 首页品牌名在顶栏和主标题同时出现，当前可接受；后续如做更强品牌视觉，可考虑把顶栏品牌压低存在感。
2. 表达训练真实角色回答在本轮浏览器测试中有一次等待接近 1 分钟；交互有等待态，未阻塞，但后续可评估更轻量的回答 prompt。
3. 正式项目移动端地图内容较长，但输入、节点、报告入口可用；后续可增加节点筛选或“当前节点优先”折叠。

## 3. 浏览器验收结果

### 3.1 首页

结果：通过。

检查点：

1. 三个入口同级可见：快速问诊、正式项目、表达训练。
2. CTA 清楚，点击分别进入 `/quick`、`/formal/new`、`/training/cases`。
3. 未出现横向溢出、按钮失效、内部词或框架错误页。

### 3.2 快速问诊

结果：通过。

真实链路：

1. 输入“线下读书会活动策划”类真实需求后进入真实会话。
2. DeepSeek 返回的首问具体自然：“主要面向哪类人群”。
3. 右侧整理区从已知内容和待确认信息开始逐步出现。
4. `123123` 这类无意义输入在入口被自然拦截，不创建模型会话。

示例链路：

1. 点击“智能海报生成网站”直接进入示例会话。
2. 示例会话显示“填入回答”，真实链路不显示，二者未混淆。
3. 理解确认阶段能点击指定卡片“制作范围”加入对话框，填入修改并发送。
4. 方案比较可见，简报页无 Markdown 原文、无内部状态词。
5. 未完成草稿保留“继续补充信息”闭环，不把缺失项写成已确认。

移动端：

1. `/quick` 首屏能看到自定义输入和第一张示例卡。
2. 会话页“对话 / 整理”切换可用，卡片引用无裁切。
3. 简报页 tab、按钮和正文无横向溢出。

### 3.3 正式项目

结果：通过，已修复报告 P2。

首页直达：

1. `/formal/new` 空提交显示自然错误：“先写一段项目描述，说明要做什么和第一版交付给谁。”
2. 填写项目描述、项目名称、相关人员、已有材料、项目约束后能创建项目。
3. 创建后进入地图工作台，表单输入被用于地图证据、当前问题和报告预览。
4. 地图从 6 项细化到 7 项，能根据主题生成活动策划模块。
5. 回答节点问题后，地图状态、待确认数和下一问都会更新。

地图交互：

1. 节点可以加入对话框，作为 compact 引用标签。
2. 未输入回答时发送禁用。
3. 发送回答后有等待态并防重复点击。
4. 报告从当前地图生成，未确认内容保留为待确认。

报告修复：

1. 项目描述、角色、材料、约束只在“已明确内容”集中出现一次。
2. 用户回答按对应问题归到正确模块：目标人群回答不再出现在活动形式模块。
3. 候选方案渲染为表格，不显示 Markdown 原始竖线。
4. 未出现 `sample/custom/partial/inferred/missing/agent/skill/API Key` 等内部词。

### 3.4 表达训练

结果：通过，已修复移动端入口 P2。

入口：

1. 固定情境覆盖学术任务、协作项目、运营指标、创意简报、外包采购、服务流程。
2. 当前版本的自定义练习不会创建回合，会用自然提示拦截。
3. 移动端已调整为先展示固定情境案例，再展示自定义输入说明。

练习：

1. 点击“提升转化率的运营诉求访谈”可进入练习。
2. “填入追问”能填入当前建议问题。
3. 角色只回答用户问到的信息，不泄露隐藏答案。
4. 输入 `123123` 时，角色提示“请换一种问法”，教练给出下一步可追问方向。
5. 提交总结后，反馈指出问到了什么、漏了什么、为什么重要、下次怎么问，并给出改写示例。

移动端：

1. 案例入口可见。
2. 训练反馈页无横向溢出。
3. “再练一次 / 完成练习”可见可点。

## 4. 报告质量检查

本轮按 Spec Kit 可借鉴思路做了产品化检查：

| 维度 | 快速问诊 | 正式项目 | 表达训练 |
|---|---|---|---|
| 完整性 | 未完成草稿保留缺失项 | 地图按模块保留待确认项 | 反馈标出漏问维度 |
| 清晰度 | 概述面向普通用户 | 详细报告分章节 | 反馈给出改写示例 |
| 一致性 | 简报来自同一 snapshot | 报告来自当前地图 | 反馈来自训练回合 |
| 可验证性 | 完成标准单独展示 | 报告含待确认和下一步 | 训练分数仅代表本轮 |
| 范围边界 | 卡片可修改范围 | 模块区分范围/风险/资源 | 不写入项目资产 |
| 未知项保留 | 未确认项不写成事实 | 待确认保留在报告中 | 漏问项明确列出 |

## 8. 发布准入结论

当前可以进入下一轮设计评审和小范围真实试用。

不建议现在投入的事项：

1. 登录注册和多人协作生产能力。
2. 正式基线发布、审批和变更生产化。
3. 大规模领域专业包。
4. SSE、微服务或工作流引擎。
