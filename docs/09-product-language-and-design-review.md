# 09 产品语言与设计评审规范

> 适用范围：需求问诊室 / Requirements Clinic 当前前端原型、交互文档、评审截图和用户可见文案。
> 当前版本：Product Language Baseline v0.2
> 更新日期：2026-07-06

本文用于把已经形成的原型、功能和文案沉淀收敛为统一口径。早期设计文案整合包已被吸收进本文和 `docs/10-release-readiness-and-design-review.md`，不再作为前端目录内的独立交付物保留；后续界面、文档和评审意见应以本文为准。

## 1. 品牌命名

### 1.1 用户可见品牌

用户可见主品牌采用中英并列：

```text
需求问诊室 / Requirements Clinic
```

使用规则：

1. 首页 H1 使用“需求问诊室”。
2. 顶部导航品牌使用“需求问诊室 / Requirements Clinic”，点击回首页。
3. 页面 metadata、工程目录、API、日志、测试和代码内部可以继续使用 `ReqClinic` 或 `Requirements Clinic`。
4. 普通页面不再只显示 `ReqClinic`，也不把“需求问诊室”写成产品副名。
5. `需求分析 Is All You Need` 是品牌口号，可以出现在首页主视觉，不替代品牌名。

### 1.2 导航顺序

所有业务页左上角统一：

```text
返回  |  需求问诊室 / Requirements Clinic
```

规则：

1. “返回”只返回上一个页面。
2. 点击品牌回首页。
3. 不使用“返回选择”“返回首页”作为左侧返回按钮文案。
4. 只有品牌按钮的无障碍标签可以写“返回首页”。

## 2. 三功能边界

| 功能 | 用户目标 | 入口 | 用户输入 | 产物 | 不能暗示 |
|---|---|---|---|---|---|
| 快速问诊 | 把一个模糊想法问清楚 | `/quick` | 一句话或选择示例 | 需求简报 | 已形成正式需求基线 |
| 正式项目 | 围绕项目建立需求地图 | `/formal/new` 或快速简报升级 | 项目描述、人员、材料、约束 | 需求地图、指导报告 | 快速问诊结论已经正式确认 |
| 表达训练 | 练习追问和总结能力 | `/training/cases` | 选择固定情境；自定义练习后续开放 | 角色回应、训练反馈 | 训练结果会写入真实项目 |

三功能必须同级出现。快速问诊不是正式项目和表达训练的唯一入口；正式项目和表达训练也不能被包装成快速问诊的样式样例。

## 3. 入口文案

### 3.1 首页

首页只解决新用户 5 秒内理解：

```text
需求问诊室
需求分析 Is All You Need
选择一个起点
```

三个模式卡片：

| 模式 | CTA | 副文案 |
|---|---|---|
| 快速问诊 | 进入快速问诊 | 一句话开始，由问诊助手连续追问并整理需求简报。 |
| 正式项目 | 进入正式项目 | 建成需求地图，围绕节点继续确认，适合协作和交付准备。 |
| 表达训练 | 进入表达训练 | 练习追问、总结，把模糊表达整理成清楚需求。 |

首页卡片不写具体案例名。案例只能在进入对应功能页后出现。

### 3.2 示例与自定义

用户可见统一使用：

| 内部概念 | 用户文案 |
|---|---|
| demo / sample | 示例体验 |
| custom | 自定义输入 |
| API Key missing | 模型服务暂未连接 |
| mock | 示例内容 / 当前版本暂未开放 |

原则：

1. 不刻意解释技术差异。
2. 只有会影响用户判断时才说明“示例不会使用你的输入”“示例不会当作你的真实项目”。
3. 自定义输入不可用时，说明能力边界并引导用户查看示例，不暴露 API Key。

## 4. 状态词映射

界面不得直接出现以下内部词：

| 禁用词 | 用户文案 |
|---|---|
| `understanding_review` | 当前理解已整理 |
| `option_review` | 方案已准备 |
| `partial` | 建议确认 / 已整理 |
| `inferred` | 初步判断 |
| `missing` | 尚未提供 / 待补充 |
| `sample` | 示例体验 |
| `custom` | 自定义输入 |
| `Mock` | 示例内容 |
| `API Key` | 模型服务 |
| 阶段 A / Phase A | 不显示；改为用户任务或能力名称 |

用户需要看到的是“下一步做什么”和“这条信息现在有多可靠”，不是系统状态机。

## 5. 三条主流程口径

### 5.1 快速问诊

定位：

```text
一句话开始，AI 连续追问并整理需求简报。
```

流程：

1. 自定义输入创建真实问诊。
2. 示例卡片直接进入独立示例会话。
3. AI 先提问，用户只回答当前问题。
4. 右侧整理内容随问答逐步出现。
5. 当前理解整理后，用户可以点击右侧卡片加入输入框并修改。
6. 方案比较后生成需求简报。
7. 简报可以继续补充，也可以升级正式项目。

### 5.2 正式项目

定位：

```text
建成需求地图，围绕节点继续确认。
```

流程：

1. 首页进入 `/formal/new`，填写项目描述、相关人员、已有材料和约束。
2. 快速问诊简报可以升级为正式项目，但只作为候选来源。
3. 工作台左侧保留对话，右侧展示可扩展需求地图。
4. 地图节点可引用进输入框，用于提出修改或补充。
5. 报告必须来自当前地图快照，不把未经确认的内容写成事实。
6. 详细报告中的项目起点、人员、材料和约束只集中展示一次；模块章节只展示该模块真正新增或确认的信息。

### 5.3 表达训练

定位：

```text
练习追问与总结，不影响真实项目内容。
```

流程：

1. 选择固定练习情境。
2. 用户练习提出追问。
3. 角色只回答用户问到的信息。
4. 教练提示下一步可以怎么问。
5. 用户提交总结后查看反馈。
6. 反馈只用于训练，不写入快速问诊或正式项目。
7. 在移动端，固定练习情境必须先于自定义输入出现，避免用户误以为当前主功能是尚未开放的自定义练习。

## 6. 报告口径

| 报告 | 面向谁 | 语言要求 |
|---|---|---|
| 需求简报概述 | 普通用户 | 说清当前理解、还需确认什么、下一步建议 |
| 需求简报详细报告 | 协作和交付对象 | 专业结构、编号清楚、范围/验收/风险完整 |
| 正式项目报告 | 项目参与方 | 章节化、可追踪、区分候选与确认 |
| 训练反馈 | 练习者 | 具体指出追问覆盖、遗漏和更好的问法 |

报告不得展示 Markdown 原文、内部状态词、模型接口细节或无依据功能承诺。

## 7. 设计评审清单

每次评审至少检查：

1. 品牌是否统一为“需求问诊室 / Requirements Clinic”。
2. 页面是否只有一个主标题，是否存在标题重复。
3. 三功能边界是否清楚，是否把示例体验误说成真实项目。
4. 自定义输入、示例体验、当前版本暂未开放三类状态是否说清楚。
5. 用户是否知道下一步点哪里。
6. 等待、空状态、错误状态是否像普通用户能看懂的话。
7. 右侧整理区或需求地图是否渐进披露，不一上来堆满信息。
8. 卡片或节点加入输入框后的反馈是否明确。
9. 桌面和 390×844 移动端是否无遮挡、无横向溢出、无死路页面。
10. 页面是否保留现有暖纸底、金色强调、衬线标题和工作台风格。

## 8. 验收标准

本规范通过的最低标准：

1. 新用户打开首页 5 秒内知道产品是“AI 引导把需求问清楚”。
2. 三个入口同级，CTA 清楚。
3. 用户可见页面不出现禁用词。
4. 示例与自定义边界真实，不承诺尚未完成的能力。
5. 快速问诊、正式项目、表达训练的主流程文案互相不混淆。
6. 概述给普通用户看得懂，详细报告保持专业。
7. 设计评审可以按本文逐项指出问题，而不是重新解释产品定位。
