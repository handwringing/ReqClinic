# 06 逐屏交互流程说明

> 本文是需求问诊室 / Requirements Clinic 当前版本的集中式交互说明。它把首页、快速问诊、需求简报、正式项目、表达训练的页面边界、进入方式、逐屏状态和关键交互统一写清楚，避免实现时把模式、示例和流程混在一起。
> 当前实现对齐日期：2026-07-07。快速问诊、正式项目和表达训练均已接真实后端链路；用户可见命名和文案口径以 [09-product-language-and-design-review.md](./09-product-language-and-design-review.md) 为准。

## 0. 设计立场

### 0.1 产品不是让用户自己写清楚需求

需求问诊室的核心不是让用户先组织完整需求，而是让用户只给出一个最小起点，然后由 AI 通过具体问题引导用户补齐关键信息。

必须坚持以下原则：

1. 用户第一句话可以很短，可以不完整，可以只是一个想法。
2. 用户第一句话来自快速问诊页的自定义输入；示例体验由固定起点直接创建独立会话，进入后续页面后不需要重复输入。
3. 后续对话由 AI 先发起具体追问，用户只回答当前问题。
4. AI 负责判断哪里还不明确，并决定下一问。
5. 右侧可视化内容随主题变化，但底层需求分析维度必须受文档约束。
6. 用户确认的是“当前理解是否符合表达”，不是正式批准。
7. 正式项目的事实、责任、证据、冲突和范围必须重新确认，不能由快速问诊结论直接替代。

### 0.2 两类分析入口必须分开

快速问诊和正式项目不是同一个流程的两种皮肤。

| 模式 | 首页入口 | 用户最小输入 | 后续主导 | 产物 |
|---|---|---|---|---|
| 快速问诊 | 首页同级模式入口：快速问诊 | 一句话想法 | AI 连续追问 | 需求简报 |
| 正式项目 | 首页同级模式入口：正式项目、快速简报升级 | 项目描述，必要时附材料、人员和约束 | AI 追问 + 人工确认 | 需求地图、指导报告 |
| 表达训练 | 首页同级模式入口：表达训练 | 选择固定练习情境 | 用户练习追问，AI 扮演角色 | 训练反馈 |

首页不再设置一个凌驾于三种模式之上的“主输入默认快速问诊”。三种模式在首页必须同级出现：快速问诊、正式项目、表达训练各自进入自己的页面或流程。快速问诊可以在入口选择后承接一句话输入，但不能让正式项目和训练看起来只是快速问诊的附属入口。

### 0.3 示例不混用

当前版本允许每个模式拥有自己的示例矩阵，但不同模式的示例数据不能互相冒充。

| 示例类型 | 所属模式 | 作用 |
|---|---|---|
| 快速问诊示例 | 快速问诊 | 展示一句话如何被 AI 追问成需求简报，并驱动右侧主题模板 |
| 正式项目示例 | 正式项目 | 展示正式项目如何记录人员、材料、约束和责任候选，并进入需求地图 |
| 表达训练示例 | 表达训练 | 让用户练习追问、归纳和规范描述项目 |

示例可以共享“需求分析”这套方法论，但不能共享结论数据。快速问诊示例不能假装成正式项目，正式项目示例也不能换标题后当作快速问诊。

当前示例覆盖：

| 类型 | 示例 |
|---|---|
| 快速问诊 | AI 海报生成网站、校园二手交易小程序、双十一护肤品投放海报、AIGC 教育影响课程论文、健身房会员续费流程、企业官网外包采购、AI 面试助手毕业设计、社恐沟通训练 AI 产品 |
| 正式项目建档 | 园区访客通行、企业官网外包、多人毕业设计 |
| 正式项目工作台 | Aster 园区访客预约与通行 |
| 表达训练 | 需求访谈、范围边界、创意简报、学术任务、服务流程、外包采购、协作项目、早期想法 |

## 1. 全局路由与页面边界

| 页面 | 路由建议 | 模式 | 页面职责 |
|---|---|---|---|
| 首页 | `/` | 分流 | 展示三个同级模式入口 |
| 快速问诊入口页 | `/quick` | 快速问诊 | 自定义输入和示例体验入口 |
| 快速问诊会话页 | `/quick/[sessionId]` | 快速问诊 | AI 追问、用户回答、右侧需求整理区 |
| 需求简报页 | `/quick/[sessionId]/brief` | 快速问诊 | 展示版本化简报、导出、反馈、升级正式项目 |
| 正式项目建档页 | `/formal/new` | 正式项目 | 收集项目起点、人员/材料候选，创建正式项目 |
| 正式项目工作台 | `/formal/[projectId]` | 正式项目 | 左侧对话 + 可扩展需求地图 + 报告投影；后台仍受正式状态机和人工关口约束 |
| 训练选择页 | `/training/cases` | 表达训练 | 选择固定练习情境；`/training` 当前重定向到该页 |
| 训练对话页 | `/training/[attemptId]` | 表达训练 | 用户练习追问，AI 扮演角色 |
| 训练反馈页 | `/training/[attemptId]/feedback` | 表达训练 | 覆盖检查、遗漏点、改进建议 |

## 2. 首页逐屏流程

### 2.1 首页目标

首页只做三件事：

1. 把快速问诊、正式项目、表达训练作为三个同级模式入口展示。
2. 点击模式后进入该模式自己的居中对话框页面。
3. 每个模式页都在顶部导航栏提供统一导航：`返回` 回上一个页面，`需求问诊室 / Requirements Clinic` 回首页。
4. 具体示例只在进入对应模式后出现，避免把示例写成首页卡片副标题。

### 2.2 首页布局

```text
┌─────────────────────────────────────────────────────────────┐
│ 需求问诊室 / Requirements Clinic                              │
│ 需求分析 Is All You Need                                      │
│                                                             │
│ 选择模式：                                                   │
│ ┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐ │
│ │ 快速问诊         │ │ 正式项目         │ │ 表达训练         │ │
│ │ 一句话连续追问   │ │ 证据与正式交付   │ │ 练习追问和总结   │ │
│ └─────────────────┘ └─────────────────┘ └─────────────────┘ │
│                                                             │
│ 点击快速问诊后进入模式页：                                   │
│ ┌─────────────────────────────────────────────────────────┐ │
│ 顶部导航：[返回] 需求问诊室 / Requirements Clinic             │
│                                                             │
│ ┌─────────────────────────────────────────────────────────┐ │
│ │ 快速问诊                                                │ │
│ │ 示例体验：AI 海报生成网站                                │ │
│ │ 我想做一个 AI 海报生成网站...                            │ │
│ └─────────────────────────────────────────────────────────┘ │
│ └─────────────────────────────────────────────────────────┘ │
│ [开始问诊]                                                  │
└─────────────────────────────────────────────────────────────┘
```

首页视觉层级要求：

1. 三个模式卡片同级、同尺寸、同权重。
2. 模式卡片的标题和副标题只描述模式价值，不把具体示例名写成卡片副标题。
3. 点击模式卡片后进入该模式自己的居中对话框页面。
4. 模式页必须在顶部导航栏提供“返回”和 `需求问诊室 / Requirements Clinic`。返回只回上一个页面；点击产品名回首页三模式选择主页。
5. 具体示例名只能在进入对应模式后出现。
6. 快速问诊输入框只出现在快速问诊模式页中。
7. 正式项目模式页进入 `/formal/new` 或正式项目示例，不先进入快速问诊。
8. 表达训练模式页进入训练选择页，不和快速问诊示例混排。

### 2.3 快速问诊入口提交

目标产品中，用户选择快速问诊后，可以输入一句话：

```text
我想做一个 AI 海报生成网站。
```

系统行为：

1. 把这句话保存为快速问诊会话的第一条用户消息。
2. 创建 `quick_session`。
3. AI 基于这句话生成第一条具体追问。
4. 跳转到 `/quick/[sessionId]`。

进入快速问诊页后，用户不再看到一个空白开场。页面已经包含：

1. 用户在首页输入的第一句话。
2. AI 的第一条具体追问。
3. 右侧根据主题初始化的需求状态卡片。

### 2.4 首页正式项目入口

正式项目入口和快速问诊入口同级。用户点击“正式项目”后进入 `/formal/new`，也可以在正式项目页查看示例工作台，而不是先进入快速问诊。

首页可出现正式项目建议 chip，例如：

```text
检测到这可能是多人协作或正式交付项目，建议创建正式项目 →
```

点击 chip 进入 `/formal/new`，并可把首页快速问诊输入作为正式项目描述草稿带入。忽略 chip 时仍停留在当前模式，不自动把正式项目降级为快速问诊。

### 2.5 首页入口规则

| 用户操作 | 进入页面 | 不允许发生 |
|---|---|---|
| 点击快速问诊 | 快速问诊入口页 | 直接显示具体示例在首页卡片副标题 |
| 点击正式项目 | 正式项目创建页 | 先创建快速问诊 |
| 点击表达训练 | 表达训练选择页 | 写入快速问诊或正式项目数据 |
| 模式页点击顶部导航栏“返回” | 返回上一个页面 | 把已进入的后续流程误清空 |
| 尝试自定义输入但未选择快速问诊 | 停留首页并提示当前入口规则 | 创建任何后续会话 |
| 快速问诊模式页点击示例入口 | 直接进入独立示例会话 `/quick/[sessionId]` | 加载正式项目数据 |
| 正式项目页点击示例入口 | `/formal/[projectId]` | 加载快速问诊简报 |

### 2.6 当前实现边界

当前版本已经不是“只能走示例”的状态。三种模式的当前边界如下：

1. 快速问诊有两条路径：自定义输入走真实后端链路；示例卡片走独立示例体验链路。
2. 快速问诊自定义输入必须先通过基本有效性检查。空输入、纯数字、乱码或明显不像需求的话，应留在入口页并给自然提示，避免浪费模型调用。
3. 快速问诊示例点击后直接进入示例会话，不再要求用户先手动把示例填入输入框。
4. 快速问诊示例会话可以提供“填入回答”“填入修改”等预设按钮；真实自定义会话不应出现这些按钮。
5. 问答过程不设置中途强制绑定卡片的回合。卡片引用只在 AI 判断当前理解已整理、用户想修改某一部分时发生；此时右侧普通模式下必须显示可选择的卡片，用户点击指定卡片后选择“加入到对话框”，再填入或输入修改内容。
6. 正式项目有两条路径：首页 `/formal/new` 建档创建项目；快速问诊简报页点击升级正式项目。两条路径都进入同一个正式项目地图工作台。
7. 表达训练当前仍以固定练习情境进入，不支持自定义训练场景；真实训练 Agent + Skill Runtime 已接入，不能把训练反馈描述为真实项目需求或能力认证。移动端必须优先显示固定练习情境，自定义输入只能作为次要记录区放在后面。
8. 当前版本不展示协议弹窗、不展示协议入口，也不展示问诊过程中的进度条或轮询条。页面状态应在操作后尽快切换；真实后端等待时只展示轻量等待反馈。
9. 正式项目报告必须从当前需求地图快照生成；项目起点、角色、材料和约束集中展示一次，模块章节只展示该模块的新增事实、推测、问题和方案。用户回答必须按被回答的问题或引用节点归因，不得贴到下一个待确认节点。

## 3. 快速问诊逐屏流程

### 3.1 快速问诊目标

快速问诊的目标是形成一份“可沟通、可执行、可继续补充”的需求简报。

快速问诊不产生正式事实、正式决策、正式基线或正式验收承诺。

### 3.2 快速问诊页面布局

```text
┌─────────────────────────────────────────────────────────────┐
│ 顶栏：返回 / 需求问诊室 / Requirements Clinic / 会话标题      │
├─────────────────────────────┬───────────────────────────────┤
│ 左栏：AI 对话                │ 右栏：需求状态                 │
│                             │                               │
│ 用户：我想做一个 AI 海报...  │ 原始想法                       │
│ AI：你希望这个网站首先...    │ 目标与使用对象                 │
│                             │ 场景与流程                     │
│ 输入框：回答 AI 的问题...    │ 功能与范围                     │
│                             │ 约束与风险                     │
│                             │ 完成标准                       │
└─────────────────────────────┴───────────────────────────────┘
```

### 3.3 进入快速问诊后的首屏

进入 `/quick/[sessionId]` 时，首屏必须已经展示首页输入和 AI 首问。

首屏示例：

```text
用户：我想做一个 AI 海报生成网站。

AI：这个网站最先要服务谁？比如你自己用来快速出图、给设计师做工具、还是开放给普通用户使用？
```

此时输入框 placeholder：

```text
回答 AI 的问题；不确定也可以说不知道。
```

禁止出现以下交互：

1. 再让用户“详细描述你的需求”。
2. 让用户自己决定接下来要补充哪些维度。
3. 只显示“我会问几个问题”，但没有具体问题。
4. 把首页输入丢失，要求用户重新输入。

### 3.4 AI 追问循环

每一轮遵循同一个循环：

| 步骤 | 页面表现 | 系统行为 |
|---|---|---|
| 1. AI 提问 | 左栏追加一个具体问题 | AI 根据当前缺口选择下一问 |
| 2. 用户回答 | 用户只回答当前问题 | 保存为当前问题的回答 |
| 3. AI 解析 | 显示“正在整理”状态 | 更新候选目标、角色、场景、需求、约束、未知 |
| 4. 右侧更新 | 卡片状态和内容变化 | 标注来源：用户明确说过、系统推测待确认、尚未提供 |
| 5. 下一问或进入复核 | AI 继续问，或生成当前理解 | 根据覆盖度和不确定性判断 |

AI 的下一问优先级：

1. 影响理解的问题，例如目标用户、使用场景、成功标准完全缺失。
2. 会影响方案方向的问题，例如是否需要账号、支付、版权、审核。
3. 会影响交付边界的问题，例如只要原型还是要完整上线。
4. 非阻断但有助于简报质量的问题，例如风格偏好、示例参考、输出格式。

### 3.5 右侧需求状态的固定约束

右侧可以根据主题换视觉组织方式，但底层必须覆盖同一组需求分析维度。

| 维度 | 快速问诊表现 | 说明 |
|---|---|---|
| 原始输入 | 原始想法卡片 | 不被 AI 改写覆盖 |
| 目标结果 | 目标/成功样子 | 用户到底想达成什么 |
| 用户/角色 | 使用者、参与者、决策者 | 不限于软件用户，也可以是客户、老师、同事、审批人 |
| 场景/流程 | 什么时候使用，怎么发生 | 需求不能脱离场景 |
| 功能/能力 | 需要系统、服务或交付物做什么 | 只记录候选，不做正式承诺 |
| 范围边界 | 做什么、不做什么 | 防止无限扩张 |
| 约束/风险 | 时间、成本、合规、技术、内容风险 | 风险不能被隐藏 |
| 方案/取舍 | 可选方案和 AI 建议 | 用户偏好不等于正式决策 |
| 完成标准 | 什么算完成、可验收 | 快速模式是沟通标准，不是正式验收 |
| 未知/待确认 | AI 判断还缺什么 | 驱动后续提问 |

### 3.6 右侧按主题呈现

右侧不是死板的六个固定卡片，可以按主题采用不同模板，但不能丢掉上面的底层维度。

| 输入主题 | 右侧推荐组织 |
|---|---|
| 软件/网站/App | 用户、场景、核心功能、数据/权限、边界、完成标准 |
| 内容/营销/设计 | 目标受众、传播目的、风格、素材、渠道、评估标准 |
| 课程/论文/作业 | 任务要求、读者/老师、论点、材料、结构、评分标准 |
| 流程/运营/服务 | 参与角色、触点流程、异常情况、责任边界、指标 |
| 采购/合同/外包 | 交付物、责任方、验收、付款/周期、风险、待确认 |
| 不明确主题 | 目标、对象、场景、输出物、约束、未知 |

### 3.6.1 右侧可视化的研究提炼

根据需求工程、目标导向需求工程、HCI / CSCW、sensemaking、信息可视化、会话式 AI、混合主导系统和认知心理学的共同结论，右侧不应被设计成摘要区，而应被设计成“需求状态机 + 共享工件 + 可操作上下文”。

右侧至少区分五类对象：

| 对象 | 页面表现 | 作用 |
|---|---|---|
| 已明确事实 | 已确认或已覆盖的卡片 | 记录用户明确表达过的内容 |
| 结构化抽取 | 当前理解、槽位、模板卡 | 把原始对话转成可讨论对象 |
| 待确认假设 | 待确认标签、未知项 | 避免 AI 把推断伪装成事实 |
| 冲突 / 风险 / 边界 | 约束风险、阻断未知、范围边界 | 暴露影响方案和交付的高价值缺口 |
| 决策与轨迹 | 方案比较、推荐理由、绑定卡片历史 | 记录为什么选、不选、撤回或修改 |

右侧展示顺序遵循：

1. 先给概述：当前理解、主题模板、状态坐标。
2. 再给结构：覆盖槽位、核心卡片、未知项。
3. 最后按需展开：主题化内容、方案比较、依据、决策理由。

这样做的原因是，需求不是一次性“收集”出来的，而是在访谈、重述、比较、假设、校正和确认中逐步共建出来的。右侧的核心价值不是“AI 已经知道什么”，而是“当前共同理解到了哪里、还缺什么、哪些是事实、哪些只是推断”。

### 3.6.2 当前右侧模板矩阵

当前快速问诊示例按以下模板运行：

| 模板 | 对应示例 | 右侧优先显示 |
|---|---|---|
| 软件构建模板 | 校园二手交易小程序 | 角色权限、核心场景、范围边界、质量约束、验收标准 |
| 创意交付模板 | AI 海报生成网站、双十一护肤品投放海报 | Creative Brief、渠道规格、风格参考、素材、审核风险 |
| 课程论文模板 | AIGC 教育影响课程论文 | 任务要求、研究问题、评分标准、证据范围、时间计划 |
| 服务流程模板 | 健身房会员续费流程 | 服务旅程、前后台责任、关键触点、失效点、指标 |
| 外包采购模板 | 企业官网外包采购 | SOW 范围、交付物、排除项、里程碑、验收与变更 |
| 多人协作模板 | AI 面试助手毕业设计 | 共同目标、角色分工、依赖关系、数据风险、决策日志 |
| 轻量灵感模板 | 社恐沟通训练 AI 产品 | 问题假设、目标用户猜测、使用时刻、替代方案、未知问题池 |

模板只改变右侧“优先展示什么”和“卡片如何组织”，不改变底层需求维度和数据隔离规则。

### 3.6.3 快速问诊不承载正式项目地图

快速问诊右侧只承载当前理解、可引用卡片、待确认信息和方案比较，不显示正式项目的分层需求地图。

原因：

1. 快速问诊的目标是形成可沟通的需求报告，不是进入完整项目治理。
2. 分层需求地图需要模块计划、责任、证据、方案分支和一致性检查，属于正式项目工作台能力。
3. 用户必须先看到报告，再主动点击“升级正式项目”，系统才重构右侧可视化。

因此，需求地图的出现时机是：

```text
快速问诊 → 查看报告 → 点击升级正式项目 → 正式项目工作台 → 地图工作台承接后续分析
```

### 3.6.4 正式项目：分层需求地图

用户从报告页点击“升级正式项目”后，正式项目不再沿用快速问诊卡片栏，而进入“左侧对话 + 地图工作台”的正式项目界面。

需求地图用于承载复杂项目的章节、模块、方案和待确认事项。它保留可视化交互特色，但层级从快速问诊的“卡片”升级为：

```text
需求地图
├─ 项目总览
├─ 模块列表
│  ├─ 目标与成功标准
│  ├─ 目标人群 / 参与角色
│  ├─ 核心流程 / 章节结构
│  ├─ 资源与约束
│  └─ 风险与预案
├─ 当前模块详情
│  ├─ 已明确
│  ├─ 系统推测
│  ├─ 待确认
│  └─ 可选方案
└─ 报告生成进度
```

扩展规则：

1. 地图节点数量由 `module_planning` 生成，不能写死为固定 4 个、6 个或 7 个。
2. 不同主题可以生成不同模块，例如活动策划可出现“活动流程、资源配置、宣发计划”，课程论文可出现“研究问题、证据范围、章节结构”，软件项目可出现“用户故事、功能需求、边界情况”。
3. 首屏只保证项目核心、当前节点、节点列表入口可见；节点过多时，节点列表内部滚动，不把整个页面撑成长页。
4. 节点标题、摘要、状态、已明确内容、系统推测、待确认问题、候选方案和关联节点都必须来自同一份地图数据，不允许只在视觉层临时拼接。
5. 地图允许后续扩展为节点分组、章节树、泳道或缩放视图，但首期至少要支持 3-12 个模块节点的稳定展示。

展示规则：

1. 首屏只显示项目总览和当前正在处理的模块，避免一次性展示完整大纲。
2. 用户打开“显示全部内容”后，才展示所有模块、待确认项和方案比较。
3. 模块状态使用自然语言：已明确、正在梳理、有方案可选、待确认、建议复核。
4. 模块内允许继续使用小卡片，但小卡片只表示事实、推测、问题或方案，不再承担整体结构。
5. 用户点击模块、问题或方案时，可以把它作为压缩引用加入对话框，继续让 AI 修改该部分。
6. AI 发现某个模块变化会影响其他模块时，右侧必须提示“相关部分也需要复核”，而不是静默改写。

按模块追问规则：

1. AI 先根据当前 snapshot 生成模块计划，而不是要求用户自己列大纲。
2. 每次只处理一个模块，左侧只问当前模块中影响最大的具体问题。
3. 对复杂模块，AI 可以生成多套方案供用户选择，例如“低成本版 / 转化优先版 / 品牌强化版”。
4. 用户可以选择方案、修改方案、暂不决定或要求 AI 再生成一组方案。
5. 方案选择记录为当前偏好，不等于正式决策。

等待页规则：

1. 生成详细指导报告必须是异步任务，不阻塞对话页。
2. 页面显示当前步骤：整理已确认内容、判断项目类型、生成模块计划、撰写章节、检查一致性、准备结果。
3. 预计时间只显示区间，例如“预计还需 1-3 分钟”，不显示虚假的精确倒计时。
4. 若生成中发现关键信息不足，应返回需要补充的问题，而不是硬写完整报告。

### 3.7 卡片点击与加入对话

右侧任何可绑定卡片都支持点击。

点击卡片后，不直接把整段内容塞进输入框，而是弹出一个悬浮小菜单：

```text
┌────────────────────┐
│ 加入到对话框        │
└────────────────────┘
```

#### 3.7.1 加入到对话框

用户点击“加入到对话框”后：

1. 小菜单关闭。
2. 输入框中出现一个压缩绑定模块。
3. 绑定模块显示卡片名称，不展开卡片全文。

示例：

```text
[范围边界]
```

用户可以继续输入：

```text
[范围边界] 这里我想先只做网页版，不做移动端。
```

建议请求结构：

```json
{
  "content": "这部分请按更保守的版本处理。",
  "bound_refs": [
    {
      "card_id": "scope-card",
      "card_type": "scope",
      "snapshot_version": 3,
      "mode": "direct"
    }
  ]
}
```

实现约束：

1. 绑定模块是输入框里的可交互模块，不是顶部状态栏。
2. 加入后只显示卡片名称，不展开卡片全文。
3. 后端必须能识别绑定的卡片版本，防止用户绑定后卡片内容变化造成歧义。
4. AI 回答时要能说明“我已根据你对功能范围卡片的修改更新理解”。
5. 卡片绑定不属于 AI 追问中途的必经步骤；不得要求用户选择普通模式下尚未显示的卡片。

### 3.8 当前理解复核

当 AI 判断继续追问的边际收益较低，或核心维度已有可用候选时，进入“当前理解复核”。

左栏 AI 展示简短理解：

```text
我现在的理解是：
1. 你想先做一个网页端 AI 海报生成工具。
2. 第一批用户是不会设计但想快速出宣传图的人。
3. 现阶段更重要的是生成速度和模板质量，不是完整商业闭环。

这份理解哪里不准确？
```

右侧卡片进入复核态：

1. 已明确的卡片显示“用户明确说过”。
2. 推测内容显示“系统推测，待确认”。
3. 缺失内容显示“尚未提供”。

用户可以：

1. 直接回答 AI 的问题。
2. 点击卡片，使用“加入到对话框”把该卡片作为修改对象。
3. 表示“不确定”，让 AI 降低结论强度并继续追问。

### 3.9 方案比较

若需求存在明显路径差异，AI 进入方案比较。

示例：

| 方案 | 适合情况 | 风险 |
|---|---|---|
| 模板优先 | 先验证用户是否需要快速出图 | 风格差异化弱 |
| AI 图像生成优先 | 视觉效果更强 | 成本、版权、审核风险高 |
| 混合方案 | 模板保证稳定，AI 提供变化 | 实现复杂度更高 |

交互规则：

1. AI 可以推荐方案，但必须说明理由。
2. 用户可以选择非推荐方案。
3. 用户选择记录为“当前偏好”，不是正式决策。
4. 如果用户的偏好与 AI 建议不一致，简报中必须同时展示差异。

### 3.10 生成需求简报

当用户确认当前理解可用，或 AI 判断核心信息足以形成简报时，进入简报生成。

页面行为：

1. AI 说明即将生成需求简报。
2. 系统创建 `brief_version`。
3. 跳转到 `/quick/[sessionId]/brief`。

## 4. 需求简报页逐屏流程

### 4.1 简报页目标

需求简报页展示同一份版本化简报的不同投影，不生成互相矛盾的新内容。

三种视图：

| 视图 | 适用场景 |
|---|---|
| 普通概述 | 给自己或他人快速理解，使用普通用户措辞 |
| 专业报告 | 作为完整需求分析文档，用于评审和后续协作，使用专业结构 |
| 导出文档 | 与专业报告同源，排版更适合复制、下载或后续转 PDF |

普通概述必须避免开发者或需求工程术语，用“现在可以这样理解 / 已经说清楚 / 建议先这样推进 / 还没完全确定的事”这类自然表达。专业报告不是“下一步清单”，而是一份可阅读的专业需求分析文档。其结构可借鉴 Spec Kit 适合的规格文档风格，至少展示：报告摘要、原始诉求与分析目标、已确认理解、用户/场景/价值、用户场景与独立验证、范围定义、关键对象、功能需求清单、成功标准与验收口径、边界情况与异常处理、假设与依赖、方案比较与推荐、风险与待确认事项、追问依据、版本说明和后续动作。导出文档只调整呈现，不新增事实。

### 4.2 简报页布局

```text
┌─────────────────────────────────────────────────────────────┐
│ 快速问诊 / AI 海报生成网站                                  │
│ [普通概述] [专业报告] [导出文档]                       [升级正式项目] │
├─────────────────────────────────────────────────────────────┤
│ 简报正文                                                     │
│ - 报告摘要、原始诉求与分析目标                               │
│ - 已确认理解、用户/场景/价值、用户场景与独立验证                │
│ - 范围定义、关键对象、功能需求清单、成功标准与验收口径           │
│ - 边界情况、假设依赖、方案比较、风险与待确认事项                │
│ - 追问依据、版本说明和后续动作                                │
│                                                             │
│ 提示：本简报用于沟通，尚未经过正式项目确认。                 │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 简报反馈

用户可以对简报反馈：

| 操作 | 行为 |
|---|---|
| 继续补充 | 回到 `/quick/[sessionId]`，保留当前简报版本 |
| 重新生成 | 基于新增回答创建新 `brief_version` |
| 导出 | 下载带版本号和正式性说明的文件 |
| 升级正式项目 | 打开升级确认流程 |

### 4.4 快速升级正式项目

升级入口位于简报页。

升级确认页必须说明：

1. 快速问诊内容会作为正式项目候选来源。
2. 快速模式的确认、完成、推荐不等于正式确认。
3. 正式项目需要证据、责任人和人工关口重新确认。

升级成功后：

1. 创建正式项目。
2. 创建 `project_intake`，记录来源 quick session 和 brief version。
3. 把快速内容迁移为候选，不迁移正式确认状态。
4. 跳转到 `/formal/[projectId]`。

## 5. 正式项目建档逐屏流程

### 5.1 正式项目为什么独立

正式项目用于真实协作、材料、责任、冲突和交付承诺。它可以从快速简报升级而来，也可以从首页直接创建。

正式项目不是让用户填写完整需求表。建档页只收集必要起点，后续仍由 AI 提问主导。

### 5.2 正式项目建档页入口

入口来源：

| 来源 | 带入内容 |
|---|---|
| 首页“创建正式项目” | 空白建档页 |
| 首页正式项目建议 chip | 把首页输入带入项目描述草稿 |
| 正式项目示例 | 进入独立示例项目，不创建你的项目 |
| 快速简报升级 | 带入 quick session 和 brief version 的候选快照 |

目标路由：

```text
/formal/new
```

### 5.3 正式项目建档页布局

```text
┌─────────────────────────────────────────────────────────────┐
│ 正式项目建档                                                 │
│ 先记录项目起点，后续由 AI 基于材料与角色继续追问。           │
├─────────────────────────────────────────────────────────────┤
│ 项目描述 *                                                   │
│ ┌─────────────────────────────────────────────────────────┐ │
│ │ 用几句话说明项目要做什么、背景是什么、第一版大致交付给谁。 │ │
│ └─────────────────────────────────────────────────────────┘ │
│                                                             │
│ 项目名称                                                     │
│ ┌─────────────────────────────────────────────────────────┐ │
│ │ 可留空，AI 可根据描述生成候选名称。                      │ │
│ └─────────────────────────────────────────────────────────┘ │
│                                                             │
│ 相关人员/角色                                                │
│ [添加角色]  角色、姓名或组织、责任、是否决策/确认人           │
│                                                             │
│ 已有材料                                                     │
│ [上传文件] [粘贴链接/文本]                                   │
│                                                             │
│ 项目约束                                                     │
│ 时间、预算、合规、技术、范围边界，可留空。                   │
│                                                             │
│ [填入建档样例]                         [创建并开始分析]     │
│ [AI 会在工作台继续追问] [建档内容只作为候选来源]             │
└─────────────────────────────────────────────────────────────┘
```

当前实现要求：

1. `/formal/new` 必须是独立页面，不从快速问诊输入框复用进入。
2. 页面主体为居中建档对话框，顶部有统一返回导航。
3. 当前版本提供示例工作台，降低用户理解成本。
4. 建档成功后直接进入 `/formal/[projectId]`，工作台左侧 AI 立即显示第一条具体追问。

### 5.4 建档字段规则

| 字段 | 是否必填 | 说明 |
|---|---|---|
| 项目描述 | 必填 | 用户只需要规范描述项目，不需要自己拆成需求点 |
| 项目名称 | 选填 | 可由 AI 生成候选，不阻断创建 |
| 相关人员/角色 | 选填但推荐 | 支持多人、多角色、责任和确认人候选 |
| 已有材料 | 选填但推荐 | 支持上传文件、粘贴材料、记录来源 |
| 项目约束 | 选填 | 时间、预算、合规、技术、范围等 |
| 身份与协议 | 后台处理 | 本地环境自动建立游客会话并完成必要同意；生产账号策略可后续收紧，但页面不以登录阻断当前体验 |

禁止出现的字段：

1. “你当前有什么困惑？”
2. “你希望 AI 怎么帮你？”
3. “请完整写出你的需求。”
4. “请一次性列出所有功能、角色、验收标准。”

这些内容由 AI 在正式项目工作台中通过追问逐步补齐。

### 5.5 建档提交行为

用户点击“创建项目并开始分析”后：

1. 校验项目描述非空。
2. 如未填写项目名称，AI 或系统生成候选项目名。
3. 创建 `project`。
4. 创建 Owner 成员。
5. 创建初始 `project_intake`，保存原始项目描述、来源、材料和角色候选。
6. 进入 `/formal/[projectId]`。
7. 右侧 AI 面板发起第一条正式项目追问。

正式项目首问示例：

```text
AI：我已经记录了项目起点。先确认分析范围：这个项目第一版交付给谁使用，最终由谁判断“可以交付”？
```

注意：这不是问用户“有什么困惑”，也不是让用户自己决定该补哪类信息，而是 AI 基于正式项目建档内容提出下一步具体问题。

当前真实链路中，提交建档会创建正式项目并启动 `formal_guidance` Job；页面进入正式项目工作台后展示地图生成或更新状态。演示链路仍可把固定案例映射为可跳转项目，但必须标明来源为示例数据。

## 6. 正式项目工作台逐屏流程

### 6.1 主布局：左侧对话 + 地图工作台

正式项目进入后，不再把旧的线性目录作为用户主导航。页面保留左侧对话框，让 AI 按当前需求地图节点连续追问；右侧是正式项目的主工作台，用地图承载模块、方案、待确认内容、影响关系和报告快照。

```text
┌─────────────────────────────────────────────────────────────┐
│ 顶栏：返回 / 需求问诊室 / Requirements Clinic / 项目名 / 地图状态 / 成员 │
├──────────────────┬──────────────────────────────────────────┤
│ 左侧：分析引导    │ 右侧：正式项目需求地图                    │
│ 当前节点摘要      │ 项目核心                                  │
│ AI 当前问题       │ 模块节点：材料、角色、目标范围、需求标准… │
│ 引用 token        │ 当前节点工作区：已明确 / 系统推测 / 待确认 │
│ 回答输入框        │ 方案选择 / 影响关系 / 报告快照             │
└──────────────────┴──────────────────────────────────────────┘
```

交互原则：

1. AI 一次只提出当前最值得确认的问题，用户只回答当前问题。
2. 用户可以点击地图节点或方案，将其作为一行内联 token 加入左侧对话框。
3. 发送后的用户气泡以卡片 token 显示引用内容，避免把结构化头部暴露给用户。
4. 每次回答都会更新当前地图节点的已明确内容、系统推测、待确认内容、方案取舍和报告快照。
5. 确认、版本、审计和变化仍受正式项目后台状态约束，但不作为页面主导航展示。
6. 顶栏“返回”统一执行返回上一个页面；点击“需求问诊室 / Requirements Clinic”回到首页。

### 6.2 进入正式项目

正式项目可以从两类路径进入：

1. 用户在正式项目页填写较完整的项目描述、材料和人员信息后创建项目。
2. 用户在快速问诊报告页点击“升级正式项目”，系统将快速问诊结果冻结为正式项目候选来源。

进入工作台后：

1. 系统先生成项目核心节点和若干模块节点。
2. 左侧 AI 直接提出第一个正式项目问题。
3. 右侧地图默认选中当前最需要处理的节点。
4. 快速问诊的结论只作为候选来源，不自动变成正式确认。
5. 示例体验可以直接跳转到项目工作台；真实后端链路仍通过创建任务和作业轮询完成。

### 6.3 地图结构

正式项目需求地图至少包含以下层次：

| 层次 | 用户看到的内容 | 作用 |
|---|---|---|
| 项目核心 | 项目名、核心目标、来源说明 | 告诉用户当前地图围绕什么展开 |
| 模块节点 | 项目起点、相关角色、目标范围、需求标准、风险取舍、报告变化 | 把复杂项目拆成可处理的分析单元 |
| 当前节点工作区 | 已明确、系统推测、待确认、可选方案 | 让用户知道当前回答会影响哪里 |
| 影响关系 | “改这里会影响”的相关节点 | 避免局部修改造成前后矛盾 |
| 报告快照 | 概述、详细报告、导出内容的同源说明 | 保证报告来自同一份整理结果 |

节点状态使用自然语言，不展示内部状态名。建议状态包括：已明确、正在梳理、有方案可选、待确认、建议复核。

地图可扩展要求：

1. 节点数量由后端模块计划决定，不是固定样例数量。
2. 3 个以内节点时，地图可以更强调当前节点和影响关系；4-8 个节点时，节点区使用自适应网格；8 个以上节点时，节点区内部滚动或分组折叠，页面整体不变成长页。
3. 不同主题可使用不同节点集合，但每个节点都必须落到统一字段：标题、摘要、状态、已明确、系统推测、待确认、候选方案、关联节点。
4. 地图只做正式项目的复杂结构展示；快速问诊继续用轻量卡片，表达训练继续用练习面板。

### 6.4 AI 主导的节点工作流

一个正式项目回合按以下方式运行：

1. Orchestrator 根据项目模式、当前节点、待确认内容和影响关系选择下一问。
2. 左侧 AI 提出一个具体问题。
3. 用户回答，或先点击地图节点/方案加入对话框再回答。
4. Structuring Skill 将回答写成结构化 patch，更新当前节点和受影响节点。
5. Validation Skill 检查完整性、清晰度、一致性、可验证性、边界和未知项。
6. 如果仍有高影响不确定项，AI 继续追问；如果已经足够清楚，进入报告快照或方案确认。

该流程不能要求用户自己列出完整大纲，也不能在中途强迫用户选择未显示的卡片或节点。

### 6.5 核心节点职责

| 节点 | 主要展示 | AI 典型追问 | 用户主要动作 |
|---|---|---|---|
| 项目起点与材料 | 原始描述、已有材料、来源可信度、缺少材料 | 哪些材料可以作为正式依据？ | 补充材料、说明来源、加入材料引用 |
| 相关角色与确认人 | 使用者、参与者、受影响方、确认人 | 最终由谁判断可以交付？ | 修正角色、指定确认人、说明责任边界 |
| 目标、范围与边界 | 要达成什么、包含什么、不包含什么 | 第一版必须包含哪些结果？ | 选择范围方案、说明排除项 |
| 需求与完成标准 | 功能候选、完成标准、验收口径 | 怎样判断这件事已经做到位？ | 修改表述、拆分需求、补充验收标准 |
| 风险、取舍与影响 | 风险、冲突、方案代价、受影响节点 | 哪些风险会影响成本、时间或承诺？ | 选择偏好、标记待确认、补充约束 |
| 报告与后续变化 | 报告结构、版本说明、变化影响 | 报告给谁看，用于评审还是交付准备？ | 生成报告、回到节点补充、查看变化影响 |

### 6.6 报告与正式确认

正式项目报告必须从同一份地图快照生成。普通概述使用用户能直接理解的措辞；专业报告使用更完整的需求分析文档结构，覆盖目标、对象、场景、范围、需求、完成标准、风险、待确认项、方案取舍和版本说明；导出文档与专业报告同源，只调整为更适合复制和后续排版的呈现。

AI 可以提出候选结论、候选方案和推荐理由，但不能替用户完成正式确认。需要责任人确认的内容必须保留“待确认”或“建议复核”状态，直到用户通过正式确认动作处理。

### 6.7 变化处理

当用户补充新材料、修改节点或提出变化时：

1. 新内容先进入左侧对话或当前节点。
2. 地图标出受影响节点。
3. 报告快照生成新版本，不覆盖历史版本。
4. 已确认内容如果受到影响，需要重新确认。
5. AI 只能说明影响和建议处理方式，不能自动沿用旧确认。

### 6.8 前端验收要求

正式项目页面必须满足：

1. 对话框固定在最左侧，地图工作台占据主要区域。
2. 不显示“项目路径”侧栏、旧线性目录、底部继续推进条或面向开发者的状态名。
3. 页面高度控制在首屏工作台内；节点数量变多时滚动节点区或当前节点区，不能出现明显长页面或大块空白。
4. 地图视觉必须清晰可见，节点、当前工作区、影响关系和报告快照不能混成普通卡片堆叠。
5. 快速问诊仍使用右侧可视化卡片；正式项目才使用需求地图，两者不能混成一个页面。
6. 所有可见词汇使用自然语言，避免直接展示 API、数据库、状态机、测试数据或内部任务名。

### 6.9 三种模式的可扩展边界

| 模式 | 可视化结构 | 适合扩展的部分 | 不应扩展成什么 |
|---|---|---|---|
| 快速问诊 | 当前理解 + 主题卡片 + 待确认信息 + 方案比较 | 模板卡片标题、优先维度、逐步披露数量、复核引用 | 不升级为正式项目地图，不要求用户管理章节树 |
| 正式项目 | 左侧对话 + 需求地图工作台 + 当前节点详情 + 报告快照 | 节点数量、节点分组、候选方案、影响关系、报告章节 | 不显示旧线性阶段目录，不把后台状态当用户导航 |
| 表达训练 | 角色对话 + 训练面板 + 情境简介 + 总结反馈 | 训练问题集、角色回答、反馈维度、情境类别 | 不写入真实项目，不展示正式项目地图 |

## 7. 表达训练逐屏流程

### 7.1 训练模式边界

表达训练是练习模式，不产生真实需求、正式事实或项目结论。

训练目标是让用户练习如何向客户、老师、同事或业务方追问，而不是让 AI 替用户完成真实需求分析。

### 7.2 训练选择页

```text
┌─────────────────────────────────────────────────────────────┐
│ 表达训练                                                     │
│ 选择一个固定情境，进入教练引导的追问练习。                   │
│ [目标澄清] [利益相关方识别] [需求工程化]                    │
└─────────────────────────────────────────────────────────────┘
```

当前实现要求：

1. `/training/cases` 是独立训练入口，`/training` 当前重定向到该页；顶部提供统一返回导航。
2. 固定情境选择区使用居中对话框，不与首页三模式卡片混排。
3. 进入练习后，训练数据只留在 `training_attempts`，不得写入快速问诊或正式项目。

### 7.3 训练对话页

```text
┌─────────────────────────────┬───────────────────────────────┐
│ 左栏：角色对话              │ 右栏：训练面板                 │
│ AI 扮演情境角色             │ 情境简介                       │
│ 教练给出当前建议追问        │ 当前训练目标                   │
│ 输入框内联情境 token        │ 总结草稿                       │
│ AI 按情境披露规则回答       │ 提交反馈                       │
└─────────────────────────────┴───────────────────────────────┘
```

训练中由用户主动提问，因为训练目的就是练习提问能力。这个模式和真实快速问诊相反，不能把训练逻辑套到快速问诊里。

但训练不能让用户空想该问什么。当前训练对话采用“教练引导下的用户追问”：

1. AI 教练显示当前建议追问，用户可点击“填入追问”放入输入框。
2. 用户可以把“情境简介”作为内联 token 加入输入框，发送后在气泡里保留 token。
3. AI 角色只回答用户已经问到的信息，不主动补全没问到的关键点。
4. 训练页不展示作业进度条；提问即时返回角色回答，总结提交后进入反馈生成。

### 7.4 训练反馈页

反馈内容：

1. 用户问到了哪些关键维度。
2. 漏掉哪些高价值问题。
3. 总结是否准确。
4. 下次可以怎么问。

训练数据不得迁移到快速问诊或正式项目。

## 8. 状态与数据规则

### 8.1 快速问诊状态

| 状态 | 页面表现 | 下一步 |
|---|---|---|
| `draft` | 已保存首页输入，等待 AI 首问 | 生成首问 |
| `clarifying` | AI 追问中，右侧状态更新 | 继续问或进入复核 |
| `understanding_review` | 当前理解复核 | 修改、继续问或进入方案比较 |
| `option_review` | 方案比较 | 选择偏好 |
| `brief_ready` | 简报可查看 | 导出、继续补充或升级 |
| `upgraded` | 已升级正式项目 | 快速记录只读 |

### 8.2 正式项目状态

正式项目状态由项目生命周期和人工关口共同决定，不与快速问诊状态互相映射。

常见状态：

1. Draft
2. Ingesting
3. Eliciting
4. Reviewing
5. Baselined
6. Reporting
7. Released
8. Changing

### 8.3 数据隔离

1. `quick_sessions` 只表示快速问诊。
2. `projects` 只表示正式项目。
3. `training_attempts` 只表示训练。
4. 快速升级正式项目时，只迁移候选和来源，不迁移正式确认。
5. 训练结果不得写入快速问诊或正式项目。

## 9. 对现有文档的补充结论

现有 PRD、FSD、数据库文档已经写到了这些原则：

1. 起始页让用户先用一句话说出当前想法。
2. AI 通过追问引导澄清。
3. 快速问诊、正式项目、表达训练是三种模式。
4. 快速问诊产物不是正式基线。
5. 正式项目需要材料、多人责任、证据、人工确认、版本和变化分析。
6. 快速升级正式项目只能迁移候选，不能迁移正式确认。

但原文档没有把以下内容写到足够可执行：

1. 首页第一句话进入快速页后如何直接触发 AI 首问。
2. 快速问诊页面首屏应显示哪些内容。
3. 正式项目和快速问诊在首页如何明确分流。
4. 正式项目自定义创建页 `/formal/new` 的具体表单和提交行为。
5. 右侧卡片“加入到对话框”的绑定细节。
6. 右侧可视化如何既主题化，又不丢失需求分析核心维度。
7. 训练模式中“用户主动提问”和真实问诊中“AI 主动提问”的区别。

本文补齐这些缺口。

## 12. 为什么之前容易偏离

之前容易出现和目标不一致，主要有四个原因：

1. 一些交互描述沿用了常见表单产品的惯性，默认让用户“描述问题、填写困惑、补充需求”，这和需求问诊室的 AI 主导问诊立场冲突。
2. 快速问诊、正式项目和训练三种模式的提问主导权不同，之前没有把边界写死，导致训练里的“用户主动追问”容易被误套到真实问诊。
3. 右侧卡片最初被理解成普通引用块，没有明确成“卡片版本 + 可选评论 + 输入框绑定模块 + 后端结构化绑定”的交互模型。
4. 正式项目入口之前只写了“项目建档”，没有把 `/formal/new` 的表单边界和首问行为具体化，所以容易被做成快速问诊的升级皮肤。

本版文档把这些点变成硬约束，后续实现和其他文档同步时应以本文为准。
